macOS Application Packaging

DMG vs PKG

Executive Summary

In the macOS ecosystem, "installing" an application isn't a single process. It varies depending on whether the app is self-contained or requires system-level integration.

  • DMG: A virtual transport container (The "Box") — simple drag-and-drop for standalone apps.
  • PKG: A procedural installer script (The "Contractor") — for complex or enterprise deployments.

The DMG (Disk Image) — The "Drag-and-Drop" Standard

Definition: A digital representation of a physical disk (similar to an ISO file).

Primary Use: Distributing self-contained applications (.app bundles).

Mechanism: Mounts as a volume (via double-click or hdiutil). User manually copies the .app to /Applications.

Key Advantages: Non-intrusive, usually no admin rights needed, clean uninstall (just delete the .app).

Focus: Aesthetics (custom background, icons, arrows) and compression.

Typical DMG drag-and-drop installation experience

The PKG (Installer Package) — The "System-Level" Specialist

Definition: An archive (XAR format) containing payload + installation instructions.

Primary Use: Complex apps needing drivers, LaunchDaemons, kernel extensions, or specific permissions.

Components:

  • Payload — actual files
  • Scripts — preinstall / postinstall (shell, Python, etc.)
  • Bill of Materials (BOM) — tracks every file & permission

Mechanism: Handled by Installer.app (GUI) or installer CLI tool.

Modern macOS Packaging

Every serious macOS distribution today must pass these three security layers:

  • Code Signing: Attach a Developer ID signature to prove the code is untampered.
  • Notarization: Submit to Apple’s service for malware scanning (required since macOS 10.14.5+).
  • Gatekeeper: macOS runtime check — blocks unsigned / unnotarized software.

Standard Installation Workflows on macOS

DMG Workflow (Simple Apps)

  1. User downloads .dmg
  2. Double-click → mounts as disk
  3. Drag .app to /Applications (or alias/shortcut provided)
  4. First launch → Gatekeeper check (notarized? signed?)
  5. App runs — done!

PKG Workflow (Complex / Enterprise)

  1. User downloads .pkg
  2. Double-click → opens Installer.app
  3. Wizard: Welcome → Readme/License → Destination → Install
  4. Admin password prompt (if needed)
  5. Pre/post scripts run
  6. Files copied, permissions set, receipts created
  7. Gatekeeper + notarization already verified

Validations & Security Checks

  • Gatekeeper — blocks execution if not signed/notarized
  • Notarization — Apple scans for malware (stapled ticket preferred)
  • XProtect + MRT — runtime malware detection
  • Quarantine flag (xattr com.apple.quarantine) — triggers Gatekeeper on first open

Rollback & Uninstallation

  • DMG-based apps: Usually just delete the .app from /Applications. No leftovers (very clean).
  • PKG-based installs:
    • Many leave receipts (/var/db/receipts/*.pkg)
    • Use official uninstaller if provided (often another .pkg)
    • Manual cleanup possible but risky (scripts may have created LaunchAgents, kernel extensions, etc.)
    • Enterprise tools (Jamf, etc.) can track & reverse changes

Best practice: Always provide clear uninstall instructions. For enterprise, prefer PKG + MDM for managed removal.

Key Takeaways for Developers & IT Admins

  • Prefer DMG for simple, standalone apps → clean user experience.
  • Prefer PKG for enterprise/MDM or apps needing background services.
  • Remember: A macOS .app is technically a bundle (special folder).
  • Tooling matters: pkgbuild, productsign, notarytool, DropDMG, Packages.app, etc.
